# Open Issues (2026-03-01)

Scope: non-scoreboard issues discovered while auditing UX/DX primitives, boundary leaks, and analysis tooling behavior.

## 1) DuckDB Feature Build Fails on Current macOS Toolchain

Severity: High

Status: Open

Summary:

Building `lab-cli` with `lab-analysis/duckdb_engine` currently fails on this machine (Apple clang 17, arm64 macOS), so DuckDB-backed commands (`views`, `query`, `trend`) remain unavailable in feature-enabled builds.

Evidence:

- Command:
  - `cargo build --manifest-path rust/Cargo.toml -p lab-cli --release --features lab-analysis/duckdb_engine`
- Failure:
  - `libduckdb-sys v0.10.2` compile error: `a template argument list is expected after a name prefixed by the template keyword`
- Relevant dependency declarations:
  - `rust/Cargo.toml` (`duckdb = { version = "0.10", features = ["bundled"] }`)
  - `rust/crates/lab-analysis/Cargo.toml` (`duckdb_engine = ["dep:duckdb"]`)

Impact:

- DuckDB-backed analysis commands cannot be delivered reliably across developer machines.
- Users receive confusing runtime behavior when expecting `views/query/trend` to work after rebuild attempts.

Recommended fix:

1. Upgrade/test `duckdb` + `libduckdb-sys` on current Apple clang toolchain.
2. Add CI job that builds `lab-cli` with `--features lab-analysis/duckdb_engine` on macOS + Linux.
3. Keep fallback error message explicit (already improved in `rust/crates/lab-analysis/src/lib.rs`).

## 2) CLI Still Leaks Internal Scheduler Concepts

Severity: Medium

Status: Open

Summary:

Public CLI output still exposes internal scheduler/state concepts that should remain implementation details if language is being aggressively standardized.

Evidence:

- `rust/crates/lab-cli/src/main.rs`
  - `rewound_to_schedule_idx` emitted in human output and JSON (`recover`).
  - Status rendering includes worker-level internals (`active_trials`, `worker_id`).

Impact:

- Users learn internal operational terms instead of stable product primitives.
- Increases coupling between UX contract and runner internals.

Recommended fix:

1. Replace scheduler-specific fields with stable outcome terms (for example: `recovered_to_safe_commit_point`).
2. Gate internal fields behind explicit debug flag (for example: `--debug-internals`).
3. Keep default UX focused on public primitives only.

## 3) Binary-Path Ambiguity Causes Inconsistent Behavior

Severity: Medium

Status: Open

Summary:

Multiple `lab-cli` binaries are present, and users may run a stale build by accident.

Evidence:

- `command -v lab-cli` -> `/Users/jevinnishioka/.local/bin/lab-cli`
- Workspace build path -> `/Users/jevinnishioka/Desktop/Experiments/rust/target/release/lab-cli`

Impact:

- Same command appears to "randomly" differ by shell/session/path ordering.
- Makes feature behavior (DuckDB or not) hard to reason about.

Recommended fix:

1. Add `lab-cli doctor` to print binary path, enabled features, and toolchain metadata.
2. Add explicit install flow (`make install-cli`) and version banner with feature flags.
3. In docs, prefer explicit binary path for local dev commands.

## 4) Public-vs-Internal Doc Boundary Not Yet Enforced Repo-Wide

Severity: Medium

Status: Open

Summary:

`README.md` was simplified, but many docs still mix public API language with patch-spec/internal implementation language.

Evidence:

- Internal-heavy docs (examples):
  - `docs/PATCH_SPEC_PARALLEL_WORKER_HARD_CUTOVER.md`
  - `docs/PARALLEL_WORKER_P0_P1_COMPLETION.md`
  - `docs/PATCH_SPEC_DURABLE_EXPERIMENT_RECOVERY.md`

Impact:

- UX/DX language remains fragmented and inconsistent across docs.
- Boundary rules become advisory instead of enforced.

Recommended fix:

1. Split docs into two explicit namespaces:
   - Public contract docs.
   - Internal engineering specs.
2. Add doc lint/check that blocks internal symbols from public docs.
3. Publish a canonical glossary and enforce one-term-per-concept.
